CI pipelines don't have time for a multi-step dance. `scalp ci` is the equivalent of `install --guarded` + `audit` + a structured report, packaged as a single command that exits 0 or 1.

## Usage

```bash
scalp ci [flags]
```

## Flags

| Flag              | Default                 | Description                           |
| ----------------- | ----------------------- | ------------------------------------- |
| `--pm`            | auto-detected           | Package manager override              |
| `--policy`        | `.scalp/policy.json`    | Policy path                           |
| `--output`        | `.scalp/ci-report.json` | JSON report path. Use `-` for stdout  |
| `--pr-context`    | `fork`                  | PR context: `fork` or `internal`      |
| `--allow-scripts` | false                   | Allow install scripts (internal only) |
| `--sarif`         | ""                      | Path for SARIF 2.1.0 report           |

## How it works

<Steps>
  <Step>**Load policy** ā€” reads `.scalp/policy.json` or uses defaults</Step>
  <Step>**Resolve** ā€” runs the package manager's lockfile-only resolve (no install)</Step>
  <Step>
    **Parse and evaluate** ā€” parses the lockfile, evaluates every package against policy rules and
    trust scores
  </Step>
  <Step>
    **Block if violations** ā€” writes the JSON report, annotates the PR (GitHub Actions), and exits
    1. Never installs.
  </Step>
  <Step>
    **Install** ā€” runs `npm install` (or equivalent) with `--ignore-scripts` in fork context
  </Step>
  <Step>
    **Hash sync** ā€” hashes every installed package directory, saves to `.scalp/lockfile.json`
  </Step>
  <Step>**Audit** ā€” verifies lockfile hashes against `node_modules`</Step>
  <Step>**Report** ā€” writes JSON report (+ optional SARIF), exits 0 or 1</Step>
</Steps>

## Basic usage

```bash
cd my-project
scalp ci
```

First run (no policy file):

```
! policy not found; allowing with audit
```

Report written to `.scalp/ci-report.json`:

```json
{
  "version": "0.2",
  "passed": true,
  "audit": {
    "verified": 142,
    "mismatched": 0,
    "missing": 0
  }
}
```

## CI with violations

If a package violates policy or trust scoring:

```
ci failed: [{"PackageID":"evil@1.0.0","Reason":"denylist","Rule":"pattern:*-free"}]
```

Report:

```json
{
  "version": "0.2",
  "passed": false,
  "violations": [
    {
      "package": "evil@1.0.0",
      "reason": "denylist_match: pattern:*-free",
      "rule": "denylist"
    }
  ],
  "audit": {
    "verified": 0,
    "mismatched": 0,
    "missing": 0
  }
}
```

## PR context

```bash
# For PRs from forks
scalp ci --pr-context fork

# For internal PRs
scalp ci --pr-context internal
```

Fork context forces `require_hash: true` and blocks install scripts. Internal context respects your policy.

## SARIF report

```bash
scalp ci --sarif .scalp/report.sarif
```

Generates a SARIF 2.1.0 report alongside the JSON report. Each violation becomes a SARIF result.

## Write report to stdout

```bash
scalp ci --output -
```

Useful for debugging or piping to other tools.

## Allow scripts (internal only)

```bash
scalp ci --pr-context internal --allow-scripts
```

<Callout type="warning">
  Allowing install scripts means packages can run arbitrary code during install. Only use this for
  trusted projects where you know exactly which packages need build steps.
</Callout>

## Why separate from `install --guarded`?

`scalp install --guarded` is a dev workflow ā€” warnings, custom PM args, policy-dependent enforcement.

`scalp ci` is a pipeline command ā€” always blocks, no passthrough args, machine-readable output.

They share the same evaluation engine. Same trust scores, same cache. The difference is the guardrails and the output format.
